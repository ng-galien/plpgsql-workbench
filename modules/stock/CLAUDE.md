# stock — Inventory Management

Articles, stock movements (in/out), multi-warehouse, alert thresholds, PMP valuation.

**Depends on:** `pgv`, `crm`

**Schemas:** `stock`, `stock_ut` (tests), `stock_qa` (seed data)

## Language Rules (STRICT)

- **Code** — ALL code in English: function names, parameter names, variable names, column names, JSON keys, comments. No exceptions.
- **Labels** — ALL user-facing text via `pgv.t('module.key')`. Never hardcode French (or any language) strings in functions. Labels live in `i18n_seed()` only.
- **CLAUDE.md** — English only.
- **Commits** — English only.
- **Examples**: `client_list` not `liste_clients`, `pgv.t('crm.action_send')` not `'Envoyer'`, `status = 'draft'` not `'brouillon'`

## pgView Framework

This module is an **independent module** of the pgView framework. Dependencies are declared in `module.json`.

### PL/pgSQL Conventions

- `get_*()` → GET pages, `post_*()` → POST actions. Naming = automatic routing via `pgv.route()`
- `nav_items() -> jsonb` → module menu. Returns `jsonb` (NEVER TABLE). Add `'entity'` key for CRUD-mapped items.
- `brand() -> text` → display name in nav
- `get_index()` → module home page (required)
- Parameters via query string: `/article?p_id=42` → `get_article(p_id int)`
- POST returns raw HTML (`pgv.toast()` + `pgv.redirect()`) — never wrapped in `page()`
- Tables via `<md>` blocks (markdown), NEVER `<table>` HTML. `<md data-page="20">` for pagination
- CSS classes `pgv-*`, NEVER inline `style="..."`
- UI primitives: `pgv.stat()`, `pgv.badge()`, `pgv.card()`, `pgv.grid()`, `pgv.empty()`, `pgv.md_table()`, `pgv.action()`, `pgv.form_dialog()`, `pgv.link_button()`, `pgv.form()`, `pgv.input()`, `pgv.sel()`, `pgv.textarea()`

### CRUD Convention

Each main entity exposes 5 standard CRUD functions consumed by `route_crud(verb, uri)`:
- `{entity}_list(p_filter text DEFAULT NULL) RETURNS SETOF jsonb` — enriched with JOINs (FK names resolved)
- `{entity}_read(p_id text) RETURNS jsonb` — full detail by id
- `{entity}_create(p_row stock.{entity}) RETURNS jsonb` — INSERT + RETURNING
- `{entity}_update(p_row stock.{entity}) RETURNS jsonb` — UPDATE by id + RETURNING
- `{entity}_delete(p_id text) RETURNS jsonb` — soft delete + RETURNING

### SDUI Convention

Each CRUD entity exposes `{entity}_ui(p_slug text DEFAULT NULL) RETURNS jsonb`:
- Without argument: list mode — returns `ui` (pgv.ui_column + pgv.ui_table) + `datasources`
- With slug/id: detail mode — returns `ui` with static components (heading, text, row, badge)

### Dev Workflow (STRICT)

1. **DDL** → Write to `build/{schema}.ddl.sql` → `pg_schema` to apply
2. **Functions** → `pg_func_set` to create/modify + `pg_test` to validate
3. **Export** → `pg_pack` (→ `build/{schema}.func.sql`) + `pg_func_save` (→ `src/`)
4. `pg_query` → SELECT/DML only, NEVER DDL or CREATE FUNCTION
5. NEVER write functions in SQL files — the workbench IS the dev tool
6. NEVER edit `build/*.func.sql` — generated by `pg_pack`

### Module Structure

- `module.json` → manifest (schemas, dependencies, extensions, sql, grants)
- `build/` → deployment artifacts (DDL + packed functions)
- `src/` → individually versioned sources (pg_func_save)
- `_ut` schemas → pgTAP tests (`test_*()`)
- `_qa` schemas → seed data only (`seed()`, `clean()`), NO pages

### DDL Content — STRICT

The DDL (`build/{schema}.ddl.sql`) contains **structure only**:

**MUST contain:** CREATE SCHEMA, CREATE TABLE, CREATE INDEX, constraints, RLS policies

**MUST NOT contain:**
- `CREATE FUNCTION` → pg_func_set then pg_pack
- `CREATE TRIGGER` → pg_pack attaches triggers to functions
- `GRANT` → pg_pack adds them in .func.sql
- `INSERT INTO` (seed data) → `build/{schema}.seed.sql` or `{schema}_qa.seed()`

### Inter-module Communication

- `pg_msg_inbox module:stock` → read incoming messages
- `pg_msg` → send a message to another module
- **feature_request / bug_report → ALWAYS via issue_report**: never send directly to another module. Create an issue: `INSERT INTO workbench.issue_report(issue_type, module, description) VALUES ('enhancement|bug', '<target_module>', '<description>')`. The lead will be notified and decide on dispatch.
- Each module is autonomous — never modify another module's functions

## i18n

The framework uses `pgv.t(key)` for internationalization. Each module must:
1. Create `stock.i18n_seed()` — INSERT INTO pgv.i18n(lang, key, value) with FR translations
2. Namespaced keys: `stock.nav_xxx`, `stock.title_xxx`, `stock.btn_xxx`, etc.
3. Use `pgv.t('stock.xxx')` in nav_items(), brand(), and all get_*/post_* functions
4. `ON CONFLICT DO NOTHING` in the seed

## QA Seed Data

The `stock_qa` schema contains only `seed()` and `clean()` — NO pages.
- `stock_qa.seed()` — INSERT realistic demo data
- `stock_qa.clean()` — DELETE in reverse FK order
- `ON CONFLICT DO NOTHING`, always use multi-tenant (`current_setting('app.tenant_id', true)`)

## Gotchas

- **tenant_id**: always `PERFORM set_config('app.tenant_id', 'test', true)` at the start of each test
- **pg_test**: discovers `test_*()` functions in the `_ut` schema
- **You are the stock agent, NOT the lead.** Never use `ws_health` to find your tasks — it shows ALL workspace tasks. Use only `pg_msg_inbox module:stock` to read YOUR messages. Only process messages addressed to `stock`.
- Movements are INSERT-only — never UPDATE/DELETE, corrections via `inventaire` movement type
- CRM must be deployed first (suppliers = `crm.client` type company)
- PMP valuation: recalculated on each entry, frozen on each exit
- Multi-warehouse: stock = SUM(movements) GROUP BY article, depot
- Alert thresholds on minimum quantity per article/depot
